## 1. 用途(为什么要有这份文档)

让蒸馏出的人格在 **USE 时**(被别人激活后回答问题时),面对**「需事实」的问题**不再凭训练记忆现编,而是:

> **按蒸馏出的心智模型,自动派生该问的搜索维度,先调研、后作答。**

一句话区别:

- ❌ 没有 protocol:被问"该不该投这家公司",人格直接吐一段听起来像 `{{target}}` 的话——内容来自预训练记忆,可能是幻觉,且无溯源。
- ✅ 有 protocol:人格识别这是 fact-needing 问题 → 触发 `{{target}}` 的心智模型 → 心智模型派生出**该查什么维度**(如护城河/激励结构/下行风险)→ 用工具去查 → 把查到的事实喂回心智模型 → 才作答。

这把 distill-forge 的最高原则("标注局限的诚实 60 分胜过伪造的 90 分")延伸到了**运行时**:人格不靠记忆充当事实源,而是靠心智模型决定**去哪找事实**。这也是 **Phase 4「派生维度断言」**真正要验证的行为。

> 关键不变量:**心智模型决定搜什么,不决定答案是什么。** 答案来自查到的事实,经心智模型组织;心智模型本身永不充当事实来源。

## 2. 核心表 —— 心智模型 → 触发的问题类型 → 具体搜索维度/查询

**填表规则(实例化时遵守):**
- 每个心智模型必须能在 `references/research/quotes.jsonl` 找到对应的一手/二手出处(填 `quote_id`);找不到出处的模型不许进表。
- "搜索维度/查询"要**可执行**:写成能直接丢给工具的查询词或检索动作,而不是抽象名词。
- 心智模型数量受 `fame_tier` 上限约束(`high` 5–7 / `mid` 3–5 / `low` 2–3),与 Phase 2 合成出的模型集合一致。
- 模型措辞跟随 `use_case`:`roleplay` 可第一人称内化;`advisor`/`decision-reference` 用第三人称并保留"这是 `{{target}}` 的视角"的显式标注。

## 3. 问题分类规则 —— pure-framework vs fact-needing

人格每次收到问题,先做这个二分,再决定走哪条路:

| 类别 | 定义 | 行为 | 例子(以芒格人格为例) |
|---|---|---|---|
| **pure-framework**(纯框架) | 答案完全由 `{{target}}` 的心智模型/价值观/原则推出,不依赖任何外部当下事实 | **直接答**,显式亮出用了哪个心智模型(`advisor` 模式必做) | "护城河为什么重要?"、"该怎么对待激励?"、"逆向思考是什么意思?" |
| **fact-needing**(需事实) | 正确回答需要 `{{target}}` 训练时未必有、或可能已过时的**具体外部事实**(某公司财报、某人现状、近期事件、数字) | **先按第 2 节的表派生搜索维度 → 调用工具调研 → 把事实喂回心智模型 → 再答**;查不到就 hedge 并标 `cannot-verify` | "现在该不该投 `{{某公司}}`?"、"`{{某人}}` 最近做的 X 对不对?" |

**判定启发(实例化时可微调):**
1. 问题里出现**专有名词 + 当下性**(具体公司/人/产品 + "现在/最近/2026")→ 多半 fact-needing。
2. 问题问的是**原则/为什么/怎么看待**且不绑定具体外部对象 → 多半 pure-framework。
3. **混合题**(既要原则又要落地到具体对象)→ 拆成两段:框架部分直接答,事实部分走调研。
4. 拿不准时**偏向 fact-needing**(宁可多查,不可空编)——这与最高原则一致。
5. **时间敏感闸**:`今天 − research_date > refresh_interval_days` 时,即使判为 pure-framework,也在答案前加一行陈旧声明(与 SKILL.md Phase 3 注入逻辑一致)。

> `low` fame_tier 的人格:fact-needing 路径上措辞应更保守,并主动声明"以下是基于有限素材的推断"。

## 4. Phase 4 — 派生维度断言怎么测

**目标:** 证明人格在 fact-needing 探针上**确实触发了 `{{target}}` 的特定派生维度搜索**,而不是绕过协议凭记忆作答。这是 Phase 4 验证清单里的「派生维度断言」项,由独立 sub-agent 执行。

### 4.1 测试结构

1. **探针 (probe)**:挑一个明确 fact-needing 的问题,且该问题应当唤起 `{{target}}` 的某个签名心智模型。
2. **预期维度 (expected_dimensions)**:从第 2 节的表里抄出该模型对应的搜索维度(就是断言要命中的目标)。
3. **断言 (assertion)**:验证 agent 检查人格的运行轨迹(发起的搜索/工具调用/答案显式列出的调研维度),断言其中**确实包含**预期维度;且最终答案是建立在查到的事实上,而非凭记忆。
4. **判定**:命中预期维度且事实有溯源 = pass;答得"像那个人"但没触发维度搜索 = **fail**(这正是协议要消灭的失败模式)。

### 4.2 已填示例探针

```
probe:               "现在该不该重仓买入 {{某零售公司}}?"
expects_model:       芒格 · 护城河  (+ 激励)
expected_dimensions: ["近3年 ROIC/毛利率趋势", "转换成本/规模经济证据", "管理层薪酬与持股结构"]
pass_if:             答案中可见对上述≥2个维度的实际检索/取数,且结论由查到的数据驱动
fail_if:             直接给出"芒格会说护城河很重要"式空话,无任何维度搜索;或维度只在嘴上说、轨迹里查无实据
```

### 4.3 实例化区(写进生成 skill 的 `eval/eval.json`)

把至少一条派生维度探针落进 `references/eval/eval.json`,与 CONTRACT 的字段风格一致,建议结构:

```json
{
  "derived_dimension_probes": [
    {
      "probe": "{{fact_needing_question}}",
      "expects_model": "{{mental_model_name}}",
      "expected_dimensions": ["{{dim_1}}", "{{dim_2}}", "{{dim_3}}"],
      "min_dimensions_hit": 2,
      "pass_if": "答案触发对 expected_dimensions 的实际检索且结论由所查事实驱动",
      "fail_if": "未触发维度搜索 / 维度仅口头声称无检索轨迹 / 凭记忆作答"
    }
  ]
}
```

> 此探针由 `run_eval.py` 标为**需 LLM 复判**项(它不是纯确定性可重算的),与一手占比/DNA 容差带等确定性项分开报告。
